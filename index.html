BIM Software Analysis · Production Stage

Revit vs Tekla
for CLT & Glulam

Comparative analysis for production-stage workflows: IFC export for fabrication, detailing, and tooling development. Context: in-house dev team available, no BTL/BTLx CNC export required.

Elements: CLT · Glulam Output: IFC for factory Dev team: 2 engineers + AI
01

Strengths & Weaknesses

Revit Autodesk
Strengths
  • +Strong API ecosystem (pyRevit, Dynamo, C# SDK) — dev team can ramp up quickly
  • +Open-source IFC exporter (IFC for Revit) — full control over output schema
  • +Flexible shared parameters — attach any fabrication data (layer sequence, grain direction, joint type) exactly as needed
  • +Best-in-class multi-discipline coordination (Arch + Structural + MEP in one ecosystem)
  • +IFC schema owned by your team — factory reads it without dependency on Tekla conventions
Weaknesses
  • −No native timber joint tools — notch, tenon, lap joint require custom families or extensions
  • −Part marking & shop drawing numbering weaker than Tekla out-of-the-box
  • ~Dev effort non-trivial: IFC Property Set Spec must be well-defined upfront to avoid rework
Tekla Trimble
Strengths
  • +Native timber component catalog: CLT panel, glulam, dowel, timber bolt ready to use
  • +Automatic numbering (assembly mark, part mark) — shop drawings production-ready natively
  • +IFC4 export includes timber attributes by default (layer orientation, board profile, lay-up)
  • +Future-proof for CNC: BTL/BTLX export native if workflow ever expands
Weaknesses
  • −IFC output uses Tekla's own schema — factory partners report inability to edit in their software
  • −Smaller API ecosystem, less documentation — deep customization harder for dev team
  • −Weak multi-discipline coordination — typically requires linking from Revit, adding a workflow step
  • ~Steeper learning curve if team is Revit-native

Implementation Roadmap

▶ Option A — Revit + Dev Recommended
1
Define IFC Property Set Spec
Document all parameters needed by the factory: layer sequence, grain direction, thickness per layer, joint type, fire rating, treatment class. Deliver as a structured table before writing any code.
Process
2
Create standardized Shared Parameter file
Map spec 1:1 to Revit shared parameters. Attach to relevant categories: Structural Framing (glulam), Floor/Wall/Roof (CLT). Use fixed GUIDs to prevent reset across projects.
Dev
3
Configure custom IFC exporter mapping
Fork or configure the open-source IFC for Revit exporter. Map shared parameters to agreed Pset_ structures. Validate output in BIMvision or Solibri before sending to factory.
Dev
4
Build Revit Add-in for data entry
In-Revit UI for filling fabrication parameters with validation (range checks, inter-layer dependency). Reduces manual error, ensures complete data before export.
Dev
5
QA pipeline + pilot project
Run one small project end-to-end: model → IFC export → factory confirms readability. Document all issues before scaling.
Process
▶ Option B — Tekla
1
Verify IFC output with the factory
Send a Tekla IFC sample, ask factory to identify exactly which fields fail and why. Distinguish schema errors from factory software limitations.
Process
2
Attempt fix via Tekla API
If the issue is property set naming or structure, Tekla .NET API can override IFC export mapping. Lower effort if only renaming/reformatting is needed.
Dev
3
Set up Revit ↔ Tekla link
Use Revit as coordination model (Arch + MEP). Link IFC or native file into Tekla for detailing. Define clear ownership: who is source-of-truth for geometry per element type.
Infra
4
Standardize numbering & mark rules
Tekla auto-numbering requires correct setup from day one (prefix, separator, sequence). Wrong rules cause re-numbering chaos when the model changes.
Process
5
Reassess after pilot
If IFC issues persist after step 2 and factory still can't read it → switch to Option A. Tekla retains advantage if a CNC workflow is activated in the future.

05

Conclusion

Short-term
Given in-house dev team + IFC as primary output + factory unable to edit Tekla IFC, Option A (Revit + custom pipeline) has the higher success probability. The team owns the output schema; factory partners are not locked into Tekla conventions.
Long-term
If BTL / BTLX export for CNC becomes necessary, or if deep timber joint detailing is required at scale, consider adding Tekla as a dedicated detailing layer linked from Revit. The two tools are complementary, not mutually exclusive.
Key risk
Option A success depends heavily on the quality of the IFC Property Set Spec defined upfront. Spec changes mid-development multiply rework cost significantly. Align with the factory on required fields and naming before writing any code.
Next actions
1. Meet with the factory — confirm exactly which IFC fields they need and in what format.
2. Write the Property Set Spec as a shared table (Excel or Notion).
3. Pilot one simple glulam element end-to-end before committing the full pipeline.
4. Assign one dev as IFC pipeline owner to maintain schema consistency.
!
The "Tekla IFC uneditable" issue may originate from the factory's software, not Tekla itself. Worth verifying before ruling out Option B entirely — if only property set renaming is needed, a Tekla API fix is faster than building the entire Revit pipeline from scratch.
