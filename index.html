Updated analysis focusing on the core problem: modeling and exporting complex timber members with fabrication-grade geometry for factory cutting. IFC must drive the machine, not just carry data.

Complex members included IFC geometry → cutting machine Revit geometry: unreliable
01

The Core Problem

Revit Geometry issues
Modeling limitations
  • −No native timber object type — must use Structural Framing, Floor, Wall, or Generic Model as workarounds
  • −Geometry engine is B-rep but not profile-based in the fabrication sense — curved members tessellate to IfcFacetedBrep on export
  • −CLT panel layer stack is visual only — no real geometry per layer in the IFC solid
  • −Notch, birdsmouth, step joint require void-cut families — boolean result unstable across instances, non-manifold geometry risk on export
  • −Join behavior between members creates solid overlaps or gaps — unreliable in IFC output
  • ~Straight rectangular glulam at simple connections: geometry acceptable but still needs verification per element
IFC export
  • −Curved members lose axis/profile data — machine cannot extract centerline, radius, cross-section
  • −Tapered beam taper parameters lost — only raw mesh geometry survives
  • −Object type mapping unreliable: CLT wall may export as IfcSlab or IfcProxy
Tekla Built for this
Modeling strengths
  • +Timber objects are native: curved member has a real centerline axis, not a hack
  • +Tapered section is a native parameter — geometry is exact, not approximated
  • +Timber joint catalog (birdsmouth, half-lap, dovetail, step joint) — built-in solid boolean, stable across instances
  • +CLT panel modeled as layered object with actual geometry per layer
  • +Member-to-member connections resolve correctly — no overlap/gap artifacts
IFC export
  • +Curved member exports with axis geometry + solid body — machine can read centerline and profile
  • +Correct IFC type mapping: IfcBeam, IfcMember, IfcPlate natively
  • +Clean solid geometry — manifold, no flipped normals from boolean operations
  • ~Schema uses Tekla conventions — factory partner needs to confirm their import workflow

Dev effort cannot fix Revit's geometry engine. Adding custom parameters or a smarter IFC exporter does not change the fact that curved members tessellate, CLT layers have no real solid, and void-cut joints are unstable. These are architectural limitations of how Revit builds geometry — not configuration problems.

06

Conclusion

Decision
Tekla is the only viable option when the requirement is fabrication-grade geometry for complex timber members — curved glulam, tapered beam, notched joints, CLT with step joint. Revit's geometry engine cannot reliably produce clean IFC solids for these cases regardless of dev investment.
Revit role
Revit remains valuable as the coordination and multi-discipline model. Timber members in Revit are spatial references only — not fabrication geometry. Tekla is source-of-truth for anything going to the factory.
IFC issue
The "Tekla IFC uneditable" complaint from the factory needs to be re-examined carefully. If Tekla geometry is clean and correct, this is likely a schema naming issue fixable via Tekla API — or a limitation of the factory's own import software. Do not let this be the reason to choose Revit for fabrication geometry.
Dev focus
Dev effort should go into: diagnosing and fixing the IFC schema issue, building any missing joint components in Tekla, automating the export/QA pipeline, and tooling the Revit ↔ Tekla coordination link — not trying to make Revit produce fabrication geometry.
Next actions
1. Export curved glulam + CLT step joint from Tekla → send to factory → get specific failure report.
2. If geometry is clean: fix schema issue via Tekla API.
3. If factory software cannot handle Tekla IFC at all: investigate cadwork or other intermediate tool as translator — still faster than rebuilding in Revit.
4. Pilot on the hardest member types before scaling.
!
If after investigation the factory truly cannot work with Tekla IFC and no fix is possible, the fallback is a geometry translation step (Tekla → cadwork → machine format) rather than remodeling in Revit. This preserves geometry integrity while solving the format compatibility problem.
