01

Strengths & Weaknesses

Revit Autodesk
Strengths
  • +Structural Framing maps to IfcBeam / IfcMember correctly when configured
  • +Team-controlled void cuts resolve joint geometry instability — solid verified before export
  • +Open-source IFC exporter — full control over property set mapping and output schema
  • +Strong API ecosystem (C#, pyRevit, Dynamo) — dev team can build tooling quickly
  • +Best multi-discipline coordination (Arch + MEP + Structural in one platform)
  • +Straight rectangular members: clean IfcExtrudedAreaSolid geometry on export
Weaknesses
  • −Curved Structural Framing: high risk of IfcFacetedBrep tessellation on export — axis and profile data lost
  • −Tapered beam: IFC geometry type depends on family construction method — needs testing
  • −CLT layer stack is parametric data only — no per-layer solid geometry in IFC
  • −No native timber joint catalog — all joint geometry requires manual void cut workflow
  • ~No path to BTL/BTLX if CNC workflow is needed later
Tekla Trimble
Strengths
  • +Curved member: centerline axis + profile preserved in IFC — machine-readable
  • +Tapered beam: native parametric solid — exact geometry, no approximation
  • +Timber joint catalog built-in (birdsmouth, half-lap, step joint, dovetail) — stable boolean, no manual void workflow needed
  • +CLT panel as layered object — real geometry per layer
  • +Correct IFC type mapping native: IfcBeam, IfcMember, IfcPlate
  • +BTL/BTLX export available if CNC workflow is needed later
  • +Auto part numbering (assembly mark, part mark) — shop drawings production-ready
Weaknesses
  • −IFC output uses Tekla schema conventions — factory partner reported inability to edit
  • −Smaller API ecosystem — dev customization harder and slower
  • −Weak multi-discipline coordination — requires Revit link for Arch/MEP
  • ~Steeper learning curve if team is Revit-native

1
Diagnose the IFC readability issue
Export curved glulam + CLT step joint from Tekla. Send to factory. Ask them to specify exactly what fails — geometry error, property set naming, or their import software limitation. These are three different problems with different fixes.
Process first

Fix IFC schema via Tekla API if needed
If issue is property set names or structure — not geometry — remap via Tekla .NET API. Targeted fix, lower dev effort than building a full Revit pipeline. If factory software itself cannot handle any Tekla IFC, investigate cadwork as a geometry translator.
Dev
3
Establish Revit → Tekla coordination link
Revit remains coordination model for Arch + MEP. Link into Tekla as spatial reference. Tekla is source-of-truth for all fabrication geometry. Define ownership boundary clearly.

Assembly mark + part mark rules defined before modeling starts. Re-numbering after model changes is expensive — wrong rules cause shop drawing chaos.

Pilot on hardest member types first
Start pilot with curved glulam and CLT step joint — not straight members. If those export cleanly and factory confirms, everything simpler will also work.

06

Conclusion

What changed
Team-controlled void cuts resolve the joint geometry problem in Revit. The scope of uncertainty is now narrow: curved glulam and tapered beam IFC geometry only. All other member types are viable in Revit + Structural Framing.
Deciding factor
A single test determines which path to take: does curved Structural Framing export as a clean parametric solid or a tessellated mesh? This is 1–2 days of work. Everything else follows from the result.
If Revit passes
Option A — full Revit pipeline. Better dev ecosystem, better coordination workflow, full schema control. Preferrable if the project has no curved members at all — commit to Revit without needing the test.
If Revit fails
Option B — Tekla primary for fabrication. Geometry is guaranteed for all member types including curved and tapered. Revit stays as coordination model. The IFC readability issue with the factory needs to be diagnosed and fixed — but this is a schema problem, not a geometry problem.
Tekla IFC issue
Do not treat this as a blocker for Option B without investigating. Clean geometry with a schema naming problem is categorically different from broken geometry. The former is fixable via Tekla API in days. Verify which one it actually is before making a decision.
Next action
Two parallel tests, both 1–2 days:
1. Revit: export curved Structural Framing + swept-blend tapered beam → check geometry type in BIMvision.
2. Tekla: export curved glulam + CLT step joint → send to factory → get specific failure report on what "uneditable" means.
Results from both tests give a clear, evidence-based decision.
